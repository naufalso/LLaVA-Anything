# LLaVa-Anything Handoff - 2026-05-19

## Current Phase

The project has just been renamed and normalized from the initial
`LLaVa-Everything` prototype to `LLaVa-Anything`.

Current status: first HF-native package skeleton is implemented and passes local
CPU/unit smoke tests. The next phase is GPU validation on an NVIDIA machine,
especially with real Qwen3/Apertus + CLIP/SigLIP configs.

`LLaVA-NeXT/` is reference-only and is ignored by this repo. Do not include it in
the GitHub project.

## Intended Architecture

The accepted architecture is documented in:

- `docs/architecture/adr-001-hf-native-composition.md`
- `docs/development/roadmap.md`

Core decision: use Hugging Face-native composition instead of LLaVA-NeXT's
class-per-LLM wrapper pattern.

The package composes:

- `AutoModelForCausalLM` language model
- `AutoModel` vision tower
- multimodal projector
- custom processor that expands `<image>` placeholders
- Auto registration for Hugging Face APIs

## Important Files

- `pyproject.toml`: package metadata, dependencies, CLI entry point
- `src/llava_anything/configuration_llava_anything.py`: `LlavaAnythingConfig`
- `src/llava_anything/modeling_llava_anything.py`: `LlavaAnythingForConditionalGeneration`
- `src/llava_anything/processing_llava_anything.py`: `LlavaAnythingProcessor`
- `src/llava_anything/builder.py`: YAML-to-HF-artifacts builder
- `src/llava_anything/auto.py`: AutoConfig/AutoModel/AutoProcessor registration
- `examples/qwen3_clip.yaml`: Qwen3 + CLIP example
- `examples/apertus_siglip.yaml`: Apertus + SigLIP example
- `tests/`: current unit and smoke tests

Start from package installation and local unit tests:

```bash
uv venv .venv

```

Build HF artifacts without model weights:

```bash

  --output-dir checkpoints/apertus-siglip-config
```

Then test full pretrained component composition. This is expected to require a

```python
    data,
```

## Minimal Inference Smoke Test

After full component loading, try:

```python
import torch
from PIL import Image
from transformers import AutoProcessor
from llava_anything.builder import load_yaml, config_from_yaml_dict, model_from_yaml_dict, processor_from_yaml_dict

data = load_yaml("examples/qwen3_clip.yaml")
config = config_from_yaml_dict(data)
processor = processor_from_yaml_dict(data, config)
model = model_from_yaml_dict(
    data,
    config=config,
    load_pretrained_components=True,
    model_kwargs={"torch_dtype": torch.bfloat16, "device_map": "auto"},
)

conversation = [
    {
        "role": "user",
        "content": [
            {"type": "image"},
            {"type": "text", "text": "Describe this image."},
        ],
    }
]
prompt = processor.apply_chat_template(conversation, add_generation_prompt=True)
image = Image.open("path/to/test-image.jpg")
inputs = processor(images=image, text=prompt, return_tensors="pt")
inputs = {k: v.to(model.device) if hasattr(v, "to") else v for k, v in inputs.items()}

with torch.inference_mode():
    output = model.generate(**inputs, max_new_tokens=64, do_sample=False)

print(processor.batch_decode(output, skip_special_tokens=True))
```

## Known Limitations

- The current implementation uses fixed image-token counts. It does not yet
  implement full LLaVA-NeXT any-resolution packing.
- Training scripts are not implemented yet.
- The projector starts randomly initialized when loading pretrained LLM/vision
  components. Meaningful responses require training or projector/adaptor weights.
- `trust_remote_code` is represented in YAML/config but should be tested with a
  real remote-code model before relying on it.
- The current processor expands `<image>` placeholders based on
  `image_seq_length`; any mismatch with the vision feature sequence length will
  raise a clear `ValueError`.
